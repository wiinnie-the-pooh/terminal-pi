# Pi Dock VS Code Extension — Developer Reference

This file documents how to build, package, publish, and version the extension.
It is the authoritative reference for both human developers and AI coding agents.

## Prerequisites

- Node.js 18+ (bundled with recent VS Code, but must be on PATH for CLI use)
- npm 9+
- `pi` CLI installed and on PATH (the tool this extension wraps)

Install dev dependencies once after cloning:

```sh
npm install
```

## Build

Compile TypeScript to `out/`:

```sh
npm run compile
```

Watch mode (recompiles on save):

```sh
npm run watch
```

`out/` is gitignored. The `vscode:prepublish` hook runs `compile` automatically before packaging/publishing, so manual compilation is only needed during development.

## Lint

Run ESLint against the TypeScript source:

```sh
npm run lint
```

The project uses an ESLint flat config (`eslint.config.mjs`) with `@typescript-eslint` rules. All source files must lint cleanly before merge.

## Test

Run the automated smoke tests:

```sh
npm test
```

Generate a code-coverage report (uses `c8`):

```sh
npm run coverage
```

Coverage reports are written to `coverage/` and include text, LCOV, and HTML output. In CI the coverage report is uploaded as a build artifact and a summary is posted to the job summary page.

For human verification in a live VS Code instance, see `TESTING.md`.

The automated test suite covers non-VS-Code helpers only; integrated terminal behavior and UX flows require manual validation.

## Continuous Integration

A GitHub Actions workflow (`.github/workflows/ci.yml`) runs on every push and pull request to `main`:

- **Matrix:** Node.js 20 / 22 on `ubuntu-latest` and `windows-latest`
- **Steps:** `npm ci` → `npm run lint` → `npm run compile` → `npm run coverage` → upload coverage artifact → `npm run package`

Ensure the workflow passes before merging. Branch protection can be configured in the GitHub repository settings to require the CI check.

## Versioning

This project uses [Semantic Versioning](https://semver.org/):

- **patch** (`0.1.x`) — bug fixes, internal refactors, no new features
- **minor** (`0.x.0`) — new commands, settings, or behaviours; backward-compatible
- **major** (`x.0.0`) — breaking changes to existing commands or settings keys

To bump the version from a **clean git working tree**:

```sh
# bump patch (0.1.1 → 0.1.2)
npm version patch

# bump minor (0.1.x → 0.2.0)
npm version minor

# bump major (0.x.x → 1.0.0)
npm version major
```

If `npm version` fails with `Git working directory not clean`, commit, stash, or discard your local changes first. `npm version` updates `package.json`, creates a git commit, and tags the commit (e.g. `v0.1.2`).

If you only want to update `package.json` without creating a commit or tag, use:

```sh
npm version patch --no-git-tag-version
```

After a normal `npm version ...` run, push the tag along with the branch:

```sh
git push && git push --tags
```

You can also edit `package.json` `"version"` manually and commit — just keep the git tag in sync.

## Package

Build a local `.vsix` installable file:

```sh
npm run package
# emits: pi-dock-<version>.vsix
```

`vsce package` reads `package.json`, compiles via `vscode:prepublish`, applies `.vscodeignore`, and writes the `.vsix`.

The `.vsix` can be distributed directly (e.g. attached to a GitHub release) without publishing to the Marketplace.

## Publish to VS Code Marketplace

Requires a publisher account on [marketplace.visualstudio.com](https://marketplace.visualstudio.com) and a Personal Access Token (PAT).

### One-time setup

1. Create a publisher at `marketplace.visualstudio.com/manage`.
2. Generate a PAT: Azure DevOps → User Settings → Personal Access Tokens → scope: **Marketplace → Manage**.
3. Authenticate locally:

   ```sh
   npx vsce login wiinnie-the-pooh
   # paste PAT when prompted
   ```

### Publish

```sh
# bump version first (see Versioning section), then:
npm run package          # verify the .vsix looks correct
npx vsce publish         # publishes current version from package.json
```

Or combine into one step:

```sh
npx vsce publish patch   # bumps patch version and publishes
npx vsce publish minor
npx vsce publish major
```

### Publish a pre-release

```sh
npx vsce publish --pre-release
```

Pre-releases are only visible to users who opt in to pre-release versions of the extension.

## Release workflow (recommended)

1. Make changes on a feature branch.
2. Run the relevant checks in `TESTING.md`.
3. Bump version: `npm version patch|minor|major`.
4. Push branch + tag: `git push && git push --tags`.
5. Open a PR; merge to `main`.
6. On `main`: `npx vsce publish` (or let CI do it).
7. Create a GitHub release pointing at the version tag; attach the `.vsix`.

## Key files for agentic editing

When adding a new command:

- `package.json` -> add entry to `contributes.commands` (and optionally `contributes.menus`)
- `src/extension.ts` -> register with `vscode.commands.registerCommand`
- `src/terminal.ts` -> add the method that builds/sends the command string
- `TESTING.md` -> add or update manual verification coverage if user-visible behavior changed

When adding a new setting:

- `package.json` -> add entry to `contributes.configuration.properties`
- `src/config.ts` -> add field to `PiConfig` interface and read it in `getConfig()`
- consumers of `getConfig()` in `extension.ts` / `terminal.ts` as needed
- `TESTING.md` -> add or update manual verification coverage if user-visible behavior changed

The compiled output in `out/` is never committed. Do not add `out/` to git.
